ProductHunt 产品信息自动采集与归档工作流
📚 案例 #001 | 日期: 2025-10-26

🎯 工作流概览

工作流目的

这是一个「ProductHunt 产品信息自动采集与归档」工作流，用于每天定时抓取 ProductHunt 上的最新产品信息，并保存为结构化的 CSV 文件。

工作流程
  1. 触发：每天纽约时间早上 9:00 自动运行
  2. 采集：从 ProductHunt RSS 订阅获取最新产品列表
  3. 处理：通过代码节点清洗、筛选、格式化数据
  4. 转换：将处理后的数据转换为 CSV 格式
  5. 存储：保存到本地磁盘进行归档

🔧 节点详细解析

1
Cron (09:00 NY)
触发器节点

⚙️ 节点功能

Cron 是一个定时触发器，用于按照设定的时间表自动启动工作流。

🎯 参数设置解析

  • 时间：09:00 - 选择早上9点是因为这是美国东部工作时间的开始，ProductHunt 的新产品通常在前一晚到早晨发布
  • 时区：NY (纽约/美国东部) - ProductHunt 是美国公司，主要用户在美国，选择纽约时区与目标市场同步
  • 频率：每天 - 保证不错过任何一天的新产品
💡 为什么要这样设置？
选择纽约时间9点的原因：
  • ProductHunt 的"Product of the Day"在美国东部时间凌晨0点重置
  • 9点运行时，前一天的产品数据已经完整
  • 避开服务器高峰期，提高采集成功率

📚 常见应用场景

  • 每日报告生成（如：每天早上8点发送昨日销售报告）
  • 定期数据备份（如：每周日凌晨2点备份数据库）
  • 周期性数据同步（如：每小时同步库存数据）
  • 定时内容发布（如：每天中午12点发布社交媒体内容）
  • 定期清理任务（如：每月1日清理过期日志）

🔄 举一反三

如果你要做其他定时任务，可以考虑：

  • 电商场景：每天凌晨1点同步订单数据到 Google Sheets
  • 社交媒体：每周一早上10点发送上周数据分析报告
  • 客户服务：每2小时检查一次未回复的工单
2
ProductHunt RSS
数据获取节点

⚙️ 节点功能

RSS Feed Reader 节点用于读取 RSS/Atom 订阅源，获取网站更新的内容。

🎯 参数设置解析

  • RSS URL：ProductHunt 的 RSS 订阅地址（通常是 https://www.producthunt.com/feed）
  • 获取数量：建议设置为获取最近24小时的产品（通常10-30条）
  • 返回格式：默认返回 JSON 格式，包含标题、描述、链接、发布时间等字段
💡 为什么使用 RSS 而不是 API？
  • 无需认证：RSS 是公开的，不需要申请 API Key
  • 稳定性高：RSS 格式标准化，很少变动
  • 足够简单：对于基本数据采集，RSS 已经满足需求
  • 免费：没有 API 调用次数限制

📊 RSS 返回的数据结构

{ "title": "产品名称", "description": "产品描述", "link": "产品页面链接", "pubDate": "发布时间", "creator": "发布者", "content": "完整内容", "categories": ["分类1", "分类2"] }

📚 RSS 节点的常见应用

  • 内容聚合：订阅多个博客/新闻源，汇总到一个地方
  • 竞品监控：监控竞争对手的博客、产品发布
  • 内容自动化：抓取内容后自动发布到自己的平台
  • 行业动态：追踪特定行业的最新资讯

🔄 举一反三

  • 博客监控：订阅竞争对手博客 RSS → 提取关键词 → 发送通知
  • 新闻聚合：订阅多个科技媒体 → 过滤关键词 → 生成日报
  • 招聘信息：订阅招聘网站 RSS → 筛选职位 → 推送到 Slack
3
Code
数据处理节点

⚙️ 节点功能

Code 节点允许你编写 JavaScript 代码来处理数据，是 N8N 中最灵活、最强大的节点之一。

🎯 在这个工作流中的作用

Code 节点可能在做以下处理：

  • 数据清洗：去除 HTML 标签、清理特殊字符
  • 字段提取：从复杂的 RSS 数据中提取需要的字段
  • 数据转换：格式化日期、处理链接、提取分类
  • 数据过滤：只保留特定类型的产品（如：AI工具）
  • 数据增强：添加额外字段，如采集时间戳

💻 典型代码示例

// 遍历所有输入项 for (const item of $input.all()) { // 提取并清洗数据 const cleanTitle = item.json.title.replace(/<[^>]*>/g, ''); const cleanDescription = item.json.description.replace(/<[^>]*>/g, '').substring(0, 200); // 格式化日期 const publishDate = new Date(item.json.pubDate).toISOString().split('T')[0]; // 提取分类 const categories = item.json.categories ? item.json.categories.join(', ') : 'N/A'; // 返回清洗后的数据 item.json = { 产品名称: cleanTitle, 产品描述: cleanDescription, 链接: item.json.link, 发布日期: publishDate, 分类: categories, 采集时间: new Date().toISOString() }; } return $input.all();
💡 为什么需要 Code 节点？
虽然 N8N 有很多内置节点，但有些特殊处理逻辑需要自定义代码：
  • 复杂的数据转换逻辑
  • 特殊的字符串处理
  • 条件判断和数据合并
  • 调用第三方库处理数据

📚 Code 节点的常见应用

  • 数据清洗：去除无用字符、统一格式
  • 数据转换：JSON 转换、日期格式化、单位转换
  • 条件逻辑：复杂的 if-else 判断
  • 数据聚合：合并多个数据源、计算统计值
  • API 交互：构建复杂的 API 请求体

🔄 举一反三

  • 价格监控：Code 节点比较新旧价格 → 计算折扣率 → 筛选优惠
  • 文本分析：Code 节点提取关键词 → 情感分析 → 分类标记
  • 数据合并：Code 节点合并客户信息 + 订单数据 → 生成客户画像
⚠️ 注意事项
  • Code 节点中使用 $input.all() 获取所有输入数据
  • 必须 return 处理后的数据
  • 可以使用大部分 JavaScript ES6+ 语法
  • 注意处理空值和异常情况
4
Convert to File (CSV)
格式转换节点

⚙️ 节点功能

Convert to File 节点将JSON 数据转换为文件格式（CSV、Excel、PDF 等），便于存储和分析。

🎯 参数设置解析

  • 输出格式：CSV - 选择 CSV 因为它通用、轻量、易于导入 Excel/数据库
  • 文件名：建议使用动态命名，如 producthunt_{{ $now.format('YYYY-MM-DD') }}.csv
  • 包含表头：通常设置为 Yes，方便后续查看和处理
  • 分隔符：默认逗号，可根据需求改为制表符或分号
💡 为什么选择 CSV 格式？
  • 通用性强：几乎所有软件都支持 CSV（Excel、Google Sheets、数据库）
  • 文件小：纯文本格式，占用空间少
  • 易于处理：可以用文本编辑器打开，方便调试
  • 导入方便：数据分析工具（Python、R）都支持 CSV

📁 文件命名最佳实践

// 推荐的动态文件命名 producthunt_2025-10-26.csv // 按日期 producthunt_2025-10.csv // 按月份 producthunt_data.csv // 单一文件（追加模式） producthunt_{{ $now.toUnixInteger() }}.csv // 使用时间戳

📚 Convert to File 的常见应用

  • 数据导出：从 API 获取数据 → 转为 Excel → 发送给团队
  • 报告生成：汇总数据 → 转为 PDF 报告 → 邮件发送
  • 数据备份：数据库数据 → 转为 CSV → 存储到云盘
  • 数据交换：系统 A 的数据 → 转为 JSON → 导入系统 B

🔄 举一反三

  • 销售报表：订单数据 → 转为 Excel → 添加图表 → 发送给管理层
  • 客户导出：CRM 数据 → 转为 CSV → 导入邮件营销工具
  • 日志归档：应用日志 → 转为文本文件 → 按日期存储
⚠️ CSV 格式注意事项
  • 如果数据中包含逗号，字段会自动加引号
  • 中文字符建议使用 UTF-8 编码
  • 日期格式要统一，建议使用 ISO 格式（YYYY-MM-DD）
  • 大数据量时考虑分批处理，避免单个文件过大
5
Read/Write Files from Disk
文件操作节点

⚙️ 节点功能

Read/Write Files 节点用于在服务器本地磁盘上读写文件，实现数据的持久化存储。

🎯 参数设置解析

  • 操作：Write File to Disk - 将数据写入磁盘
  • 文件路径：建议使用绝对路径，如 /data/producthunt/
  • 文件名：从上一个节点获取（已包含日期）
  • 写入模式：
    • Overwrite：覆盖现有文件（每天生成新文件）
    • Append：追加到文件末尾（累积所有数据）
💡 文件存储策略建议
根据不同需求选择存储方式：
  • 每日一个文件：便于按日期查找，文件管理清晰
  • 单一大文件：便于做长期趋势分析，但文件会越来越大
  • 按月归档：平衡文件大小和管理便利性

📂 推荐的目录结构

/data/ └── producthunt/ ├── 2025-10/ │ ├── producthunt_2025-10-01.csv │ ├── producthunt_2025-10-02.csv │ └── producthunt_2025-10-26.csv ├── 2025-11/ └── archive/ └── producthunt_2024.csv

📚 文件操作节点的常见应用

  • 数据归档：定期保存重要数据到本地备份
  • 日志记录：将工作流运行日志写入文件
  • 缓存数据：暂存中间处理结果
  • 配置管理：读取配置文件驱动工作流
  • 报告生成：保存生成的报告文件

🔄 举一反三

  • 发票备份：从邮件提取发票 → 保存到本地 → 按月归档
  • 图片下载：抓取网页图片 → 保存到磁盘 → 按分类存储
  • 配置驱动：读取 JSON 配置 → 动态调整工作流行为
⚠️ 文件操作注意事项
  • 权限问题：确保 N8N 有目录写入权限
  • 磁盘空间：定期清理旧文件，避免磁盘满
  • 路径安全：避免使用用户输入直接构建路径（防止目录遍历攻击）
  • 并发写入：多个工作流同时写入同一文件可能导致数据损坏
  • 云端考虑：如果在云环境运行，考虑使用对象存储（S3、Google Cloud Storage）

🚀 工作流优化建议

💡 可以添加的功能

1. 错误处理

  • 添加 Error Trigger 节点捕获失败情况
  • 失败时发送通知到 Slack 或邮件
  • 记录错误日志到文件

2. 数据去重

  • 在 Code 节点中添加去重逻辑
  • 读取昨天的文件，对比已有产品
  • 只保存新增的产品

3. 数据增强

  • 调用 ProductHunt API 获取更多信息（点赞数、评论数）
  • 使用 AI 节点分析产品描述，提取关键词
  • 添加产品分类标签

4. 通知功能

  • 筛选出高热度产品（>100 赞）
  • 发送到 Slack/Email/微信
  • 生成每日摘要报告

5. 数据可视化

  • 将数据同步到 Google Sheets
  • 使用 Google Data Studio 制作仪表板
  • 追踪产品趋势和分类统计

⚡ 性能优化

  • 限流控制：如果数据量大，添加延迟避免被封 IP
  • 分批处理：使用 Split In Batches 节点处理大量数据
  • 缓存机制：使用 Redis 缓存已处理的产品 ID
  • 压缩存储：定期将 CSV 压缩为 ZIP 节省空间

📝 学习总结

核心知识点
  1. 定时触发：使用 Cron 节点实现定时任务，注意时区选择
  2. 数据采集：RSS 是获取公开数据的简单有效方式
  3. 数据处理：Code 节点提供最大灵活性，用于复杂逻辑
  4. 格式转换：将数据转为通用格式（CSV）便于后续使用
  5. 数据存储：本地文件存储适合数据归档和备份
工作流设计思维
  • 🎯 明确目标：先确定要解决什么问题
  • 📥 数据输入：数据从哪里来？（API、RSS、数据库、文件）
  • ⚙️ 数据处理：需要什么转换？（清洗、筛选、聚合、计算）
  • 📤 数据输出：数据去哪里？（存储、发送、展示）
  • 🔄 循环迭代：不断优化和完善工作流
